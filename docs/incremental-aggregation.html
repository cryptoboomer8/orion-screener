Incremental Aggregation

Why re-processing the whole archive every 5 minutes is a waste of CI time, bandwidth, and battery — and how to stop doing it without changing anything the dashboard sees.

Full vs Incremental

O(1) · flat forever

What the aggregator actually touches

Click the mode toggle above to see the difference. In full mode every historical snapshot lights up — each one is re-read on every trigger. In incremental mode only the newest file matters; the rest stay dim because we already captured them last time.

Why this scales differently

Assumes 5-minute Worker cadence (~8,700 snapshots per month). Incremental time is read + parse of one file; full time scales with total history.

How it actually works

a

How do we know which files are new?

The existing snapshots.json is already sorted by timestamp. Read the last row’s ts — anything newer in snapshots/ is new. No separate state file.

latest_ts = rows[-1]["ts"]
b

What if a snapshot lands late or out of order?

Safety net: compare against the max timestamp in current output. Stragglers with older timestamps are skipped (already captured or would break ordering). Worker commits are monotonic in practice.

if file_ts > latest_ts: process
c

What about momentum.json?

No incremental needed — it’s always computed from the single latest .json.gz. Same as today. The speedup is entirely on snapshots.json, where the linear cost used to live.

leaderboard(latest_gz.tickers)

Trade-offs to eat

Nothing here is free. Two things get slightly more complex:

The aggregator now has two code paths

  • “Cold start” when snapshots.json doesn’t exist — runs full.
  • “Incremental” when it does — appends only new rows.

Full rebuilds are occasionally needed

  • Scoring formula changes, new fields, or bugs in earlier rows all require reprocessing history.

Mitigation

Add a --full flag: python scripts/aggregate_snapshots.py --full reprocesses everything. CI defaults to incremental. One-off full runs stay one command away.

What this unlocks

5-minute cadence becomes free
Triple the capture rate without any aggregator-side cost. Momentum verdicts update ~10 min sooner on fast movers.
Aggregator always runs in <100ms
Whether you have one week or one year of history. CI timeouts stop being a concern.
Room for heavier per-snapshot analytics
Rolling per-coin volatility, cohort stats, regime detection — all fine to add when each run only touches one file.
R2 migration ports cleanly
Incremental logic is source-agnostic. When raw storage moves to R2, “list files in git” becomes “list objects in R2” — the rest stays the same.
